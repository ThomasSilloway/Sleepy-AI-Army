# CTO Critique of Error Handling Solutions (V2 Brainstorming)

This document provides a critique of the three solutions presented in `brainstorming_v2.txt`, makes a recommendation, and outlines a refined A+ solution. The user's feedback emphasized a preference for simpler try-catch blocks for individual operations, or context managers that are not overly verbose upfront, and an interest in an outer handler or declarative state system.

## 1. Critique of V2 Brainstormed Solutions

### a. Refined Context Manager (`NodeOperationContext`)

*   **Pros:**
    *   Encapsulates error handling logic (exception catching and non-exception `signal_error`) and state updates for a defined block of operations.
    *   `signal_error()` method is a direct answer to handling non-exception error conditions within the block.
    *   Can make the `with` block itself cleaner for the happy path.
    *   State updates are centralized within the context manager's `__exit__` method.
*   **Cons:**
    *   The context manager's `__exit__` method can become very complex, containing extensive logic for different error types and state updates (similar to the initial `handle_operation_error` utility).
    *   Flow control outside the `with` block still requires manual checks of `state['error_message']` (or a similar flag) to decide if the node should return, which is something we aimed to simplify.
    *   While `signal_error` is good, it's an explicit call that developers must remember. If they raise an exception instead, or if a non-covered exception occurs, the handling might be less specific than desired unless `__exit__` is very sophisticated.
    *   The upfront setup of `custom_failure_updates` for the context manager instance can be verbose if many specific state changes are needed for that block.
*   **Grade: C+**
    *   **Justification:** It addresses the `signal_error` request but centralizes a lot of logic into `__exit__`, potentially recreating the complexity of a single large utility function. The manual flow control check after each `with` block is not ideal. It's an improvement over basic context managers but doesn't fully align with simplifying the node's main path decision-making on errors.

### b. Simplified Try-Catch with Custom Exceptions

*   **Pros:**
    *   Aligns well with the user's preference for explicit `try...except` blocks around fine-grained operations.
    *   Custom exceptions (`LLMOperationError`, `FileWriteError`) provide semantic meaning to errors.
    *   The `process_node_error` function centralizes the *translation* of these custom exceptions into `WorkflowState` updates, which is good.
    *   Reduces boilerplate *within* the main `try` block of the node, as individual `except` blocks just re-raise a specific custom error.
*   **Cons:**
    *   Requires defining and maintaining a potentially large hierarchy of custom exceptions if we want fine-grained error types.
    *   The inner `try...except...raise CustomError(...)` pattern, while simple, is still repeated for every operation.
    *   The `process_node_error` function can become a large dispatcher, with many `isinstance` checks to determine how to update state based on the custom exception type. This can mirror the complexity of the initial `handle_operation_error` utility if not managed carefully.
    *   The node still needs a top-level `try...except` to call `process_node_error` and then `return state`.
*   **Grade: B-**
    *   **Justification:** This pattern is quite good and directly addresses the user's preference for try-catch. The custom exceptions are a strong point. However, the repetitive nature of inner try-catch-raise blocks is a form of boilerplate itself, and the central processor might become complex. It's a solid contender but could be more streamlined in how the processing is invoked.

### c. Declarative State Outcome Configuration

*   **Pros:**
    *   Highly centralizes how `WorkflowState` is modified based on predefined outcomes, making state update rules very explicit and configurable per node.
    *   Node logic can focus on performing operations and raising appropriate custom exceptions (tying in well with approach #2).
    *   Reduces direct state manipulation boilerplate *within* the node's primary logic flow.
    *   `NODE_STATE_OUTCOMES_CONFIG` provides a clear, readable map of error types/outcomes to state changes.
    *   Template-based error messages and summaries offer good consistency.
*   **Cons:**
    *   The `NODE_STATE_OUTCOMES_CONFIG` dictionary can become very large and potentially hard to manage across many nodes and diverse error types.
    *   Requires a robust mechanism (decorator or a very disciplined `try...finally` structure in each node) to ensure `finalize_node_state` is always called with the correct `NodeOutcome`. This mechanism itself could be complex or add boilerplate if not a decorator.
    *   Less flexible for highly dynamic error messages or state updates that don't fit simple templates or flag settings.
    *   Initial setup and maintenance of the configuration.
*   **Grade: B**
    *   **Justification:** This is a strong conceptual approach, especially for standardizing state updates. It aligns with the "outer handler" idea. The main challenges are the practicalities of managing the configuration and ensuring the `finalize_node_state` function is invoked correctly and consistently. It pairs well with custom exceptions.

## 2. Recommendation (from V2 Brainstorming)

I recommend a hybrid approach that primarily leverages **Simplified Try-Catch with Custom Exceptions (Approach #2)** for identifying and classifying errors within the node, combined with a refined version of the **Declarative State Outcome Configuration (Approach #3)** for processing these errors and updating the state. The mechanism for invoking this processing should ideally be a **decorator**.

**Justification:**

*   **User Preference Alignment:** This combination directly supports the user's preference for try-catch blocks around operations. Custom exceptions make the "what went wrong" clear.
*   **Centralized State Logic:** The declarative configuration (from Approach #3) provides a clean way to manage *how the state should change* for each *type* of custom error, avoiding a massive `if/elif` block in the `process_node_error` function from Approach #2.
*   **Readability & Maintainability:** Node logic becomes: `try -> operation -> except -> raise CustomError`. The state update rules are defined separately in a config, which is maintainable. The decorator handles the outer try-catch and processing call.
*   **Reduced Boilerplate:**
    *   Individual operation blocks are simple: `try-except-raise`.
    *   The top-level try-catch in the node is eliminated by the decorator.
    *   The final state update logic is handled by a central utility driven by the declarative config, removing repetitive state field setting from the node's main error handling path.
*   **Effectiveness:** This combination should be effective. The node clearly signals *what* error happened (via custom exception), and a central, configurable system determines the *consequences* for the state.

## 3. Generate Final A+ Solution (from V2 Brainstorming)

**Concept: Custom Exceptions with Declarative Handler via Decorator**

This A+ solution refines the hybrid of V2 approaches #2 and #3.
1.  **Custom Exception Hierarchy:** Define a clear, minimal set of custom exceptions (e.g., `LLMError`, `FileError`, `ChangelogError`, `GitError`, `ValidationError`, `NodeConfigError`) that inherit from a common `NodeOperationError`. These exceptions will carry the original error, an operation type (enum), and any relevant context.
2.  **Node-Level Try-Except-Raise:** Nodes will use simple `try...except` blocks around individual operations. If an operation fails (either via exception or a return code indicating failure), the node will raise the appropriate custom exception, including the `NodeOperation` enum value.
3.  **Node Decorator (`@handle_node_exceptions`):** A decorator will wrap the entire node function. This decorator is responsible for:
    *   A single, top-level `try...except` block within the wrapper.
    *   Catching our defined `NodeOperationError` subtypes.
    *   Catching generic `Exception` as a fallback.
    *   Calling a `_process_error_and_update_state` utility, passing the `state`, `config` (from the wrapped function's arguments), and the caught exception.
    *   Ensuring the `state` is returned after error processing.
4.  **`_process_error_and_update_state` Utility with Declarative Config:**
    *   This utility (internal to the module providing the decorator) takes the `state`, `config`, and the `exception`.
    *   It looks up a declarative configuration (e.g., `NODE_ERROR_CONFIG` dictionary) keyed by the `NodeOperation` enum value found in the custom exception (or a default like `NodeOperation.UNEXPECTED` for generic exceptions).
    *   This configuration defines:
        *   `summary_template` (e.g., `"LLM operation failed: {specific_message}"`).
        *   A dictionary of `state_updates` (e.g., `{"is_manifest_generated": False, "manifest_data": None}`).
        *   Optional: `set_aider_exit_code` (boolean, defaults to True).
    *   The utility performs logging (using `exc_info=True` and the original exception if wrapped), formats messages using templates and exception details, and updates the `WorkflowState` as per the configuration.
    *   It also handles appending to `state['error_message']` (using `f"[{operation_type.value}] {specific_message}"`) and setting `state['aider_last_exit_code']` consistently.

**How it Mitigates Cons / Incorporates Strengths:**

*   **Reduces Node Boilerplate:** The decorator completely removes the top-level `try...except...return state` boilerplate from the node code. Inner try-catch blocks are minimal (`try: op(); except LibError as e: raise MyCustomError("message", original_exception=e)`).
*   **Centralized but Configurable Logic:** State update rules are centralized in the declarative `NODE_ERROR_CONFIG`, not a giant `if/elif` in a processing function. This makes it easy to see and modify how errors affect state for each `NodeOperation` type.
*   **Readability:** Node code focuses on its core operations and raising specific, meaningful errors. The decorator transparently handles the consequences. Error impacts are clearly defined in the config.
*   **Maintainability:** Custom exceptions are defined in one place. State update rules are in the `NODE_ERROR_CONFIG`. The `_process_error_and_update_state` utility and decorator are generic.
*   **User Preferences Met:**
    *   Nodes use simple `try-except-raise` for operations.
    *   The decorator acts as the "outer handler."
    *   The `NODE_ERROR_CONFIG` is the "declarative state system."
    *   Context managers are avoided for this primary error handling flow, keeping individual operation handling explicit.

# --------- src/utils/custom_exceptions.py ---------
from enum import Enum

class NodeOperation(Enum): 
    LLM_CALL = "LLM Call"
    FILE_WRITE = "File Write"
    CHANGELOG_UPDATE = "Changelog Update" 
    GIT_COMMIT = "Git Commit" 
    VALIDATION = "Validation"
    CONFIGURATION = "Configuration"
    UNEXPECTED = "Unexpected"

class NodeOperationError(Exception):
    def __init__(self, message: str, operation_type: NodeOperation, original_exception: Exception = None, **kwargs):
        super().__init__(message)
        self.message = message 
        self.operation_type = operation_type
        self.original_exception = original_exception
        self.additional_data = kwargs

    def __str__(self):
        return self.message

class LLMError(NodeOperationError): 
    def __init__(self, message, original_exception=None, **kwargs):
        super().__init__(message, NodeOperation.LLM_CALL, original_exception, **kwargs)

class FileError(NodeOperationError): 
    def __init__(self, message, original_exception=None, **kwargs):
        super().__init__(message, NodeOperation.FILE_WRITE, original_exception, **kwargs)

class ChangelogError(NodeOperationError): 
    def __init__(self, message, original_exception=None, **kwargs):
        super().__init__(message, NodeOperation.CHANGELOG_UPDATE, original_exception, **kwargs)

class GitError(NodeOperationError): 
    def __init__(self, message, original_exception=None, **kwargs):
        super().__init__(message, NodeOperation.GIT_COMMIT, original_exception, **kwargs)
        
class ValidationError(NodeOperationError):
     def __init__(self, message, original_exception=None, **kwargs): 
        super().__init__(message, NodeOperation.VALIDATION, original_exception, **kwargs)

class NodeConfigError(NodeOperationError): 
     def __init__(self, message, original_exception=None, **kwargs):
        super().__init__(message, NodeOperation.CONFIGURATION, original_exception, **kwargs)

NODE_ERROR_CONFIG = {
    NodeOperation.LLM_CALL: {

    },

    NodeOperation.CONFIGURATION: {
        "summary_template": "Node configuration error: {specific_message}",
        "state_updates": {"is_manifest_generated": False, "manifest_data": None}
    },
    NodeOperation.CHANGELOG_UPDATE: {
        "summary_template": "Changelog update failed: {specific_message}",
        "state_updates": {"is_changelog_entry_added": False}
    },
    NodeOperation.GIT_COMMIT: {
        "summary_template": "Git commit failed: {specific_message}",
        "state_updates": {}, 
        "set_aider_exit_code": False 
    },
    NodeOperation.UNEXPECTED: { 
        "summary_template": "An unexpected error occurred: {specific_message}",
        "state_updates": { 
            "is_manifest_generated": False, "manifest_data": None,

        }
    }

def _get_logger(state: dict, node_config_arg: dict, default_name: str = 'NodeExceptionHandler'):
    # Use 'node_config_arg' which is the 'config' dict passed to the node
    if node_config_arg and "logger" in node_config_arg and node_config_arg["logger"] is not None:
        return node_config_arg["logger"]
    return logging.getLogger(state.get('current_step_name', default_name))

def _process_error_and_update_state(state: dict, node_config_arg: dict, exception: Exception):
    logger = _get_logger(state, node_config_arg) 
    
    operation_type = NodeOperation.UNEXPECTED
    error_config_details = NODE_ERROR_CONFIG[NodeOperation.UNEXPECTED] 
    specific_message = str(exception)
    exception_for_trace = exception

    if isinstance(exception, NodeOperationError):
        operation_type = exception.operation_type
        error_config_details = NODE_ERROR_CONFIG.get(operation_type, NODE_ERROR_CONFIG[NodeOperation.UNEXPECTED])
        if exception.original_exception:
            exception_for_trace = exception.original_exception
    
    error_log_message = f"[{operation_type.value}] {specific_message}"
    summary_for_state = error_config_details.get("summary_template", "Operation failed: {specific_message}").format(specific_message=specific_message)

    logger.error(error_log_message, exc_info=exception_for_trace) 

    existing_error_message = state.get('error_message', "")
    if existing_error_message and not existing_error_message.endswith('\n'):
        existing_error_message += "\n"
    state['error_message'] = f"{existing_error_message}{error_log_message}"
    state['last_event_summary'] = summary_for_state

    if "state_updates" in error_config_details:
        for key, value in error_config_details["state_updates"].items():
            state[key] = value
            
    apply_exit_code = error_config_details.get("set_aider_exit_code", True)
    if apply_exit_code and state.get('aider_last_exit_code') is None:
        state['aider_last_exit_code'] = -1

def handle_node_exceptions(node_function_to_wrap):
    @wraps(node_function_to_wrap)
    def wrapper(state: dict, config: dict, *args, **kwargs): 
        try:
            return node_function_to_wrap(state, config, *args, **kwargs)
        except NodeOperationError as noe:
            _process_error_and_update_state(state, config, noe) # Pass the node's 'config'
            return state 
        except Exception as e: 
            # Wrap unexpected errors for consistent processing
            unexpected_error = NodeOperationError(str(e), NodeOperation.UNEXPECTED, original_exception=e)
            _process_error_and_update_state(state, config, unexpected_error) # Pass node's 'config'
            return state
    return wrapper

This A+ solution provides a robust framework by:
*   Defining clear, semantic custom errors that include the `NodeOperation` type.
*   Allowing nodes to focus on their core logic and raising these specific errors.
*   Using a decorator to transparently handle these custom errors (and any unexpected generic exceptions) at the node boundary.
*   Employing a declarative configuration (`NODE_ERROR_CONFIG`) keyed by `NodeOperation` to map error types to specific state updates (messages, summaries, flags), ensuring consistency and maintainability.
This significantly reduces boilerplate within the node functions themselves regarding error handling and state manipulation for error conditions.
